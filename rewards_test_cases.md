# 0Buck 奖励系统测试用例 (QA Test Suite v1.0)

本用例供 **程序猿** (DID-DB9653-765527) 和 **Shopify-专家** (DID-0D58EF-FA849A) 在集成联调时进行业务规则校验。

---

## 1. 签到返现模块 (Check-in)

| ID | 场景描述 | 预期结果 |
| :--- | :--- | :--- |
| TC-01 | 首次签到激活 | 订单 `fulfilled` 后次日 00:00，`checkin_plans` 状态转为 `active_checkin`，第一天签到成功。 |
| TC-02 | P1 连续签到完成 | 连续签到 5 天后，自动发放 5% 奖励至 `wallets`，`current_period` 转为 2。 |
| TC-03 | **签到中断顺延** | P1 第3天漏签，第4天点击签到。系统应跳过 P1，直接将第4天记为 **P2 的第1天**。P1 奖励不可再领取。 |
| TC-04 | 555天到期结算 | 到达 555 天时，若 P20 进行到第10天（未满30天），系统自动按 `(10/30)*5%` 计算残余奖励并清算，计划标记为 `completed`。 |

## 2. 拼团免单模块 (Group-buy)

| ID | 场景描述 | 预期结果 |
| :--- | :--- | :--- |
| TC-05 | 模式二选一 | 用户选择拼团模式后，原有签到计划应立即标记为 `active_groupbuy`，不可再进行每日签到。 |
| TC-06 | 拼团达标触发 | 分享产生第3个支付订单（且通过退货期）后，系统自动触发 100% 返现至用户余额。 |
| TC-07 | 拼团失败退款 | 拼团有效期内，若其中一个订单退款导致不满3人，拼团计划状态转为 `expired`，余额不予返还。 |

## 3. 推荐分润模块 (Referral)

| ID | 场景描述 | 预期结果 |
| :--- | :--- | :--- |
| TC-08 | 实时分润锁定 | 被推荐人下单支付，推荐人 `wallet_transactions` 出现一笔 `status: pending` 的记录。 |
| TC-09 | 订单取消解冻失败 | 被推荐人订单取消/退款，推荐人的 `pending` 记录状态转为 `failed`。 |
| TC-10 | 等级自动晋升 | 推荐人 2 年内累计结算总额超过 $1,000，下一笔订单奖励比例自动从 1.5% 升至 2.0%。 |

## 4. 极端异常与安全 (Edge Cases)

| ID | 场景描述 | 预期结果 |
| :--- | :--- | :--- |
| TC-11 | 余额负值处理 | 用户领取奖励后退款，系统扣回余额。若余额不足，`wallets.balance_available` 应允许出现负值，并在下次奖励中抵扣。 |
| TC-12 | Metafields 一致性 | 后端 `wallets` 余额更新后，Shopify `customer.metafields.0buck_rewards.wallet_balance` 必须在 1 分钟内完成同步。 |
