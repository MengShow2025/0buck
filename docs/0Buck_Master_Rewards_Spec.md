# 0Buck 奖励与分销系统全规格说明书 (Master Rewards Spec)

## 1. 系统概览
0Buck 奖励系统旨在通过高粘性（签到）、高爆发（拼团）和长尾收益（粉丝分销）构建增长闭环。系统包含六大核心支柱，并由一套**优先级引擎**驱动结算。

---

## 2. 六大核心支柱

### 支柱 1：签到返现 (Sign-in Cashback)
*   **触发条件**：订单确认收货且过退货期。
*   **周期**：500天内。
*   **结构**：分为 20 期发放，总额等于"奖励基数"（实付金额 - 运费 - 税费）。
*   **多订单处理 (Batch Check-in v3.4.7)**：
    *   用户每日只需执行一次“注入”操作，即可为当天所有活跃订单完成签到。
    *   每个订单独立核算进度、周期与奖励。
    *   前端智能展示最近达成里程碑/结算周期的订单。
*   **算法**：
    *   **P1-P2 (引导期)**：固定（4%, 6%），周期短（5d, 10d）。
    *   **P3-P20 (长期期)**：每期 25-30 天。包含常规期（3-5%）与爆点大奖期（8%, 10%）。
        *   **爆点出现机制（先小后大）**：采用分段隔离抽取算法。
            *   **P3-P10（前期）**：仅从普通奖励池抽取，爆点概率**比较低**。
            *   **P11-P20（后期）**：将剩余普通期与 8%、10% 两个爆点混合打乱，爆点出现概率**明显更高**。
        *   **权威算法参考（JavaScript）**：
            ```javascript
            /**
             * 0Buck 船长精简版生成器
             * 核心：P1-P2固定，P3-P20线性决定是否触发爆点池
             */
            const _ = require('lodash');

            function generate0BuckFinalRoadmap() {
                // 1. 初始 20 个组合全家桶
                const p1_p2 = [{ r: 4, d: 5 }, { r: 6, d: 10 }];
                const big_hits = [{ r: 8, d: 30 }, { r: 10, d: 30 }];
                const normal_30d = [
                    { r: 5, d: 30 }, { r: 5, d: 30 }, { r: 5, d: 30 }, { r: 5, d: 30 }, { r: 5, d: 30 }
                ];
                const normal_25d = [
                    { r: 5, d: 25 }, { r: 5, d: 25 }, { r: 5, d: 25 }, { r: 5, d: 25 },
                    { r: 3, d: 25 }, { r: 4, d: 25 }, { r: 4, d: 25 }, { r: 4, d: 25 },
                    { r: 4, d: 25 }, { r: 4, d: 25 }, { r: 4, d: 25 }
                ];

                // 2. 核心：分段隔离抽取
                // 前期 (P3-P10)：只从 16 个普通里抽 8 个，且优先排 30 天的磨损期
                let all_normals = _.shuffle([...normal_30d, ...normal_25d]);
                let early_stage = all_normals.slice(0, 8);
                let remaining_normals = all_normals.slice(8);

                // 后期 (P11-P20)：剩下的普通 + 两个大爆点，全部打乱
                let late_stage = _.shuffle([...remaining_normals, ...big_hits]);

                // 3. 最终流水线
                return [...p1_p2, ...early_stage, ...late_stage].map((item, index) => ({
                    phase: index + 1,
                    ratio: item.r,
                    days: item.d
                }));
            }
            ```
*   **断签规则**：
    *   任何一期中断，本期奖励作废。
    *   下次签到从**下一期**的第一天开始。
*   **续签机制**：每期可使用 1 次"续签卡"（积分兑换），最大断签天数需 ≤ 5天。

### 支柱 2：拼团购 (Group Buy / Share-to-Free)
*   **入口**：
    *   **订单详情页**：针对已购买的特定商品发起分享。
    *   **已购商品详情页**：针对已购买商品生成的分享链接。
*   **触发条件**：通过该特定商品分享链接产生的新订单。
*   **免单规则 (Progressive Settle Rule)**：
    *   **阶梯结算**：每满 3 人购买，系统立即结算 1 件商品免单。
    *   **数量要求**：若购买了 N 件商品，则总共需要 `N * 3` 人购买来实现全部免单。
    *   **最后一件**：最后一件商品的免单条件同样是 3 人，但若受邀人数超过 `N * 3`，仍按 N 件免单计算（不溢出）。
*   **抉择期与 P1 冲突**：
    *   仅在 **第一期签到 (Phase 1 / 5天内)** 且未领取返现前可选。
    *   一旦领取 P1 返现或进入 P2，拼团免单资格永久失效。
*   **优先级**：低于“分销奖”，高于“推荐奖励/粉丝奖”。

### 支柱 3：用户分销 (User Direct Distribution)
*   **入口**：通过 **“商品专属链接”**（Product/Merchant Share Link）产生的一次性购买。
*   **触发条件**：单次交易驱动，不限新老用户。
*   **奖励额度 (v3.4.7)**：
    *   **Silver (3.0%)** / **Gold (4.0%)** / **Platinum (5.0%)**。
*   **优先级**：**最高优先级**。触发此奖励后，不计算拼团或粉丝奖励。

### 支柱 4：用户推荐奖励 (User Fan Referral)
*   **入口**：通过 **“固定专属注册链接”**（Registration Link）吸引新用户注册。
*   **机制**：建立 2 年 LTV 粉丝绑定关系。
*   **触发条件**：粉丝在 2 年内产生的 **非专属商品链接** 订单。
*   **奖励额度 (v3.4.7)**：
    *   **Silver (1.5%)** / **Gold (2.0%)** / **Platinum (3.0%)**。
*   **优先级**：**最低优先级**（保底）。若订单已触发分销或拼团，则不再发放推荐奖励。

### 支柱 5 & 6：达人分销与达人推荐奖励 (KOL System)
*   **身份一致性**：达人与用户逻辑完全相同，仅比例不同。
*   **准入与谈判**：达人需申请入住，比例由平台根据谈判结果最终达成并记录在 `dist_rate` 和 `fan_rate` 字段。
*   **默认比例范围**：
    *   **达人分销**：8% - 20% (Negotiated)。
    *   **达人推荐奖励**：3% - 8% (Negotiated)。

## 3. 优先级引擎逻辑 (Priority Engine)
当一个订单涉及多个推荐关系或满足多个奖励条件时，系统按以下优先级 **仅发放一种** 奖励，且各档比例随用户/达人等级动态调整：

1.  **达人分销 (KOL Distribution)**: 8% - 20%（优先级最高）。
2.  **用户分销 (User Direct Distribution)**: 3% - 5%（优先级次之）。
3.  **拼团免单 (Group Buy)**: 3人免单奖励（由用户主动选择触发）。
4.  **达人粉丝奖励 (KOL Fan Rewards)**: 3% - 8%（2年LTV）。
5.  **用户粉丝奖励 (User Fan Rewards)**: 1.5% - 3%（2年LTV）。

*注：*
*   *同一笔订单只产生一种分销/推荐奖励。*
*   *用户自身的"签到返现"是基础权益，除非用户选择了"拼团免单"，此时二者互斥。*

## 4. 结算与提现规则 (Settlement & Payout Rules)

### 4.1 核心原则 (Post-Return Period Settlement)
为确保财务安全，0Buck 遵循 **“有效完成订单 (Signed + 15D)”** 的严格结算原则：
*   **启动时机**：订单进入“有效完成”状态后的 **第二天**，系统才正式开启该订单的 500 天签到计划。
*   **异常挂起**：若发生退货、换货或物流纠纷，签到计划立即 **“停止并等待 (Suspend & Wait)”**，直到重新签收并度过新的退货期后，方可继续进度。
*   **奖励冻结**：
    *   **非交易积分**：行为完成即时发放（受日上限约束）。
    *   **交易类积分/奖励**：产生时即进入 `frozen` 状态，仅在订单“有效完成”后解冻。
    *   **穿仓处理**：若退款时积分已消费导致余额不足，无需向用户追回（积分价值较低，以用户体验为主）。

### 4.2 自动追回机制 (Clawback Logic)
若订单在有效完成前发生退款：
*   **资金扣除**：自动从受益人钱包中扣除 `pending` 状态的佣金与返现。
*   **状态作废**：该订单对应的计划立即转为 `forfeited`。

## 5. 达人 (KOL) 奖励深度逻辑

### 5.1 分销奖 (Distribution)
*   **逻辑**：纯链接驱动，不动（不分享/无成交）则无收益。

### 5.2 推荐奖励 (Referral / Fan Bond)
*   **有效期**：**严格 2 年期 (LTV)**。
*   **淘汰机制**：粉丝绑定关系在 2 年后自动解除，变回“路人”。
*   **增长压力**：达人与用户必须持续补充新用户进入网络，以抵消 2 年期满后的粉丝流失。

---

## 5. 用户与达人等级系数
各奖励比例区间（如用户分销 3%-5%、KOL分销 8%-20%）均与**用户等级**或**达人等级**挂钩，等级越高系数越大。具体等级划分及对应系数由管理后台配置（非硬编码），可随运营策略动态调整。

## 6. 积分系统与续签卡 (Points & Renewal Card)

### 6.1 积分获取规则 (Points Earning)

#### 6.1.1 非交易类奖励 (Daily Cap: 150 Pts)
为保持系统平衡，所有非交易行为产生的积分每日累计上限为 **150 Pts**：
*   **涵盖种类**：正常签到、写帖子、写评论、参与活动、参与预售/众筹、提建议等。
*   **结算时机**：行为完成即时发放（受日上限约束）。

#### 6.1.2 交易类奖励 (No Cap)
交易驱动的积分奖励无上限，但遵循 **“有效完成订单”** 原则：
*   **结算前提**：必须满足“签收 + 过退货期”的条件。
*   **奖励基数**：
    *   **自己交易**：`订单实付金额 * 5`。
    *   **粉丝交易**：`订单实付金额 * 2`。
*   **特性**：仅销售类积分与订单的有效性强绑定，若发生退款则按比例追回。

### 6.2 续签卡 (Renewal Card) 规则
*   **兑换价格**：**3000 Pts / 张**。
*   **使用限制**：每期仅限使用 1 次，断签后 5 天内必须使用。

## 7. 提现、财务与收入 (Withdrawal & Revenue)

### 7.1 提现手续费 (Withdrawal Fees)
*   **虚拟货币 (USDC/USDT)**：**$2 / 每次** (固定费用，**平台强烈鼓励使用**，手续费最低)。
*   **其他方式 (如 PayPal)**：**3.2% + $2 / 每次**。
*   **扣除方式**：所有手续费均直接从用户的 **提现金额内扣除**。
*   **最小提现金额**：**$50 USD**。

### 7.2 汇率管理逻辑 (Exchange Rate Management)
系统遵循三阶段汇率换算原则，以确保财务安全与成本对冲：

1.  **采购入库阶段 (CNY $\rightarrow$ USD)**：
    *   从 1688 等平台抓取商品时，原始人民币 (CNY) 价格进入服务器时立即按系统汇率换算为 **美元 (USD) 成本**。
    *   此阶段应用 **+0.5% 的汇率对冲 Buffer**。

2.  **前端展示阶段 (USD $\rightarrow$ Local Currency)**：
    *   服务器中的 USD 价格根据用户的地理位置/偏好币种，实时换算为 **本地货币 (如 EUR, GBP, JPY)** 展示给用户。

3.  **支付结算阶段 (Local $\rightarrow$ USD Payment)**：
    *   用户以本地货币金额支付时，支付网关/系统将其换算回 **美元 (USD) 支付金额** 进行最终财务结算。

### 7.3 平台收入
*   **来源**：汇率对冲产生的 Buffer (0.5%)、未兑付的准备金、以及提现手续费差价（若有）均归为 **平台收入**。

## 10. 零信任安全架构 (v3.5.0)

### 10.1 身份认证与访问控制 (Auth & AuthZ)
*   **JWT 核心化**：废弃基于 Cookie 的 Session 模拟，全站 API 强制校验由后端签发的加密 JWT。
*   **强制 IDOR 防护**：所有涉及个人数据的接口（钱包、订单、计划）强制执行 **“资源所有权校验”**，禁止通过修改 URL 中的 ID 越权访问他人数据。

### 10.2 逻辑安全防护 (Business Logic Security)
*   **时区跳跃防御**：签到逻辑统一采用 **UTC 零点** 作为结算基准，限制用户修改时区的频率（每 30 天仅限一次）。
*   **AI 安全沙箱**：Dumbo AI 的工具调用链中增加权限拦截层，严禁 AI 跨用户查询敏感内存或财务事实。

### 10.3 财务数据完整性 (Financial Integrity)
*   **复式记账对账**：系统每日凌晨自动执行 **“日终对账脚本”**。
    *   校验逻辑：`当前余额 == 初始余额 + Σ(收入流水) - Σ(支出流水)`。
    *   异常处理：若对账不平（Diff != 0），系统立即自动锁定相关账户提现功能并触发管理员报警。
*   **行级悲观锁**：涉及资金扣减的操作强制使用 `.with_for_update()`，彻底杜绝高并发下的“双花”漏洞。

## 8. 数据库模型建议
*   **Referrals表**：记录 `inviter_id`, `invitee_id`, `expiry_date`, `type (USER/KOL)`。
*   **RewardPhases表**：记录每个订单的 20 期执行状态。
*   **Points表**：记录用户积分，用于兑换"续签卡"。
*   **UserLevels表**：记录用户/达人等级及对应奖励系数，支持后台动态配置。
